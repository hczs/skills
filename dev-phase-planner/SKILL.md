# Dev Phase Planner

基于项目的 PRD、技术选型、系统架构（以及可选的接口规范）文档，制定 **MVP 优先**的分阶段 TDD 开发计划。

## 核心原则

1. **MVP 优先**：先交付能跑通核心流程的最小系统，再逐步完善
2. **每个阶段必须可独立验收**：产出的代码不是"残废"，用户可以实际操作并看到效果
3. **TDD 驱动**：每个阶段的开发遵循 红→绿→重构 循环
4. **渐进式**：后续阶段依赖前面阶段的产出，不能跳跃

## 工作流程

### 第一步：读取项目文档

找到并阅读以下文件（按优先级查找，不要假设路径）：

1. **PRD**（产品需求文档）：`PRD.md`、`prd.md`、`docs/PRD.md`、`docs/prd.md` 等
2. **技术选型**：`tech-stack.md`、`技术选型.md`、`docs/tech-stack.md` 等
3. **系统架构**：`architecture.md`、`架构设计.md`、`docs/architecture.md` 等
4. **接口规范**（可选）：`docs/plans/*-api-spec.md` — 如果已通过 `pages-to-api-spec` 完成 UI 验证，会有这份接口契约文档。**如果存在则直接使用**

如果找不到（除接口规范外），主动询问用户提供路径。读取后梳理：
- 核心功能列表（从 PRD 提取）
- 技术栈约束（语言、框架、数据库等）
- 系统模块划分（从架构提取）
- 外部依赖（第三方服务、API 等）

### 第二步：分析并规划 MVP

先回答这几个问题（写在输出的总览文档中）：

1. **核心用户流程是什么？** — 用户从进入到完成核心价值的完整路径
2. **哪些功能是"必须有"才能跑通这个流程的？** — 这些是 Phase 1 的内容
3. **哪些功能是"有了更好"但不影响核心流程的？** — 排到后续阶段
4. **哪些功能可以后期再加？** — 排到最后或标记为 future

MVP 的判断标准：**用户能走完核心流程，系统能响应，数据能流转。** 不需要好看，不需要高性能，不需要处理所有边界情况。

### 第三步：划分阶段

将项目拆分成多个阶段（通常 3-7 个），每个阶段遵循以下规则：

**阶段必须独立可验收**：
- 完成后，用户可以启动应用/服务并实际操作
- 有明确的验收标准可以手动验证
- 不依赖"未来阶段"的代码才能运行

**阶段粒度建议**：
- Phase 1（MVP）：核心流程能跑通 — 最小但完整
- Phase 2-N：逐步增强（扩展功能、UI 完善、性能优化等）
- 每个阶段的预计工作量保持相对均匀

**关于前端验证**：如果用户已通过 `pages-to-api-spec` 完成了 UI/UX 验证并产出了 `api-spec.md`，planner 会直接使用该接口规范，Phase 1 可以专注于后端 API + 前后端联调。如果没有接口规范，则需要在 Phase 1 中同时处理前端骨架和后端逻辑。

**TDD 融入每个阶段**：
- 每个阶段内部的任务按 TDD 顺序排列
- 先写测试 → 再写实现 → 验证通过 → 重构

### 第四步：输出文档

在项目的 `docs/plans/` 目录下输出。

#### 文件命名规则

```
docs/plans/{prd名称}-{日期}-{功能点}.md
```

- `prd名称`：对应 PRD 文档的名称（去掉扩展名），如 `todo-app`
- `日期`：计划创建日期（不是最后更新日期），格式 `YYYYMMDD`，如 `20260510`。迭代更新不改文件名
- `功能点`：该阶段的核心功能关键词，如 `mvp`、`auth`、`categories`
- 示例：`todo-app-20260510-mvp.md`、`todo-app-20260510-categories.md`

如果一个 PRD 拆分出多个阶段，在同一批次中统一 prd 名称和日期，只改功能点后缀。

#### 1. 总览文档 `docs/plans/{prd名称}-{日期}-总览.md`

```markdown
# [项目名] 开发计划

## MVP 分析
- 核心用户流程：[描述]
- MVP 必须功能：[列表]
- 技术栈：[关键约束]

## 阶段总览

| 阶段 | 文件 | 里程碑 | 依赖 |
|------|------|--------|------|
| Phase 1 | [链接] | [验收点] | 无 |
| Phase 2 | [链接] | [验收点] | Phase 1 |
| ... | ... | ... | ... |

## 验收路线图
[每个阶段完成后如何验证的简要说明]
```

#### 2. 每个阶段一个 MD 文件 `docs/plans/{prd名称}-{日期}-{功能点}.md`

```markdown
# Phase N: [阶段名称]

## 当前进度

- [x] Task 1: [任务名]
- [ ] Task 2: [任务名]
- [ ] Task 3: [任务名]
- [ ] Task 4: [任务名]

> 从第一个 `[ ]` 开始执行。完成后将 `[ ]` 改为 `[x]`。

## 里程碑目标
[一句话描述这个阶段完成后系统是什么状态]

## 验收标准
- [ ] 验收项 1：[具体操作步骤 + 预期结果]
- [ ] 验收项 2：...
- [ ] 验收项 3：...

## 完成条件
- 本阶段所有验收标准全部通过
- 所有 Task 的 checkbox 已全部 `[x]`
- 核心流程能手动走通一遍

## 前置依赖
- 需要完成 Phase X
- 需要的外部服务/配置：[列出]

## 任务拆解（TDD 流程）

### Task 1: [任务名]
**目标**：[这个任务做什么]

**测试用例**：
- test_[功能]_[场景]_[预期结果]
- ...

**实现要点**：
- [需要创建/修改的文件]
- [关键实现思路]

**验收**：[这个任务完成后怎么手动验证]

### Task 2: ...

## 涉及文件
| 操作 | 文件路径 | 说明 |
|------|----------|------|
| 新建 | src/xxx/xxx.ts | ... |
| 修改 | src/xxx/xxx.ts | ... |

## 阶段完成后的系统状态
[描述完成后的系统能力，强调用户可以做什么]

## 已知限制（本阶段及近期都不处理的）
- [列出整个项目中明确不做、或远期才做的功能。本阶段不做但下一阶段会做的功能，不放这里，放在下一阶段的目标中说明]
```

## 输出要求

- 每个阶段的 MD 文档控制在 **100-200 行**，保持可读性
- 验收标准必须是**可手动操作验证的**，不能是"代码能编译"这种
- 每个阶段的最后要说明"完成后的系统状态"，让用户知道下一步能做什么
- 如果一个阶段太大（超过 10 个任务），考虑拆分成两个阶段
- Phase 1 必须是最小但完整的，不能是"建了个项目框架"这种空壳

## 进度追踪机制

每个阶段的 MD 文档是**可执行的工作文档**，LLM 通过顶部 checkbox 追踪进度。

### LLM 执行协议

当 LLM 被要求执行某个阶段的 plan 时：

1. **读取文档**：先看「当前进度」区，找到第一个 `[ ]` 的 Task
2. **执行 TDD 流程**：按该 Task 的测试用例 → 实现 → 验证顺序执行
3. **更新进度**：完成一个 Task 后，将对应的 `[ ]` 改为 `[x]`，然后继续下一个
4. **阻塞处理**：如果某个 Task 尝试 3 次仍无法通过，将其标记为 `[x]` 并在该 Task 下方加 `> blocked: [原因]`，跳到下一个可独立完成的 Task

### 进度更新示例

执行过程中，LLM 直接修改该 MD 文件的 checkbox：

```markdown
## 当前进度

- [x] Task 1: 创建项目结构
- [x] Task 2: 实现 Todo CRUD API
- [ ] Task 3: 实现前端页面
- [ ] Task 4: 添加数据持久化
```

## 边界情况处理

- **PRD 不够详细**：基于已有信息做合理推断，在文档中标注为"待确认"
- **架构文档缺失**：根据技术选型推断模块划分，标注为建议方案
- **项目太小**（只有一个核心功能）：可以只有 1-2 个阶段，不必强行拆分
- **项目太大**（超过 7 个阶段）：考虑将后面的阶段标记为"后续迭代"，只细化前 3-4 个阶段

## 示例

输入：一个 Todo App 的 PRD + React + Node.js + SQLite 技术选型

输出结构：
```
docs/plans/
├── todo-app-20260510-总览.md        # MVP 分析 + 阶段总览
├── todo-app-20260510-mvp.md         # Phase 1: 核心 CRUD + 联调
├── todo-app-20260510-categories.md  # Phase 2: 分类功能
├── todo-app-20260510-auth.md        # Phase 3: 用户认证
└── todo-app-20260510-polish.md      # Phase 4: UI 优化 + 部署
```

> 如果已有 `api-spec.md`（由 `pages-to-api-spec` 产出），planner 会参考接口规范来制定计划，Phase 1 可专注于后端 API + 联调。

Phase 1 的验收标准示例：
- [ ] 启动应用后可以看到 Todo 列表页面（此时是真实数据）
- [ ] 可以添加一个新 Todo，输入标题后点击提交
- [ ] 可以勾选完成一个 Todo
- [ ] 可以删除一个 Todo
- [ ] 刷新页面后数据仍然存在
