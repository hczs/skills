# Pages to API Spec

分两步走：先指导用户用 impeccable 完成页面设计，再根据页面产出接口规范文档。

## 工作流程

### 第一步：指导用户使用 impeccable

先读取项目信息，然后告诉用户该怎么用 impeccable：

1. **读取 PRD**：找到 `PRD.md`、`prd.md`、`docs/PRD.md` 等，了解有哪些页面和交互
2. **读取技术选型**：找到 `tech-stack.md` 等，了解前端用什么框架
3. **检查 PRODUCT.md**：impeccable 需要这个文件来了解产品定位

然后给用户一个清晰的操作指南，告诉他们接下来该做什么。格式如下：

---

**接下来你需要用 `/impeccable` 完成页面设计。以下是具体步骤：**

**1. 如果是新项目（没有 PRODUCT.md）**，先让 impeccable 了解你的产品：

```
/impeccable teach
```

它会问你产品定位、用户群、品牌调性等问题，回答即可。

**2. 逐页面设计**，对每个页面调用：

```
/impeccable 为 [页面名] 设计页面。

数据需求：
- [字段名]: [类型] — [说明]

需要包含的交互：
- [交互 1]
- [交互 2]
```

**你项目中需要设计的页面**：[根据 PRD 列出具体页面]

**3. 设计完所有页面后**，用浏览器实际走一遍核心流程，确认交互没问题。如果需要实时调整，可以用：

```
/impeccable live
```

**4. 页面确认后**，回来告诉我"页面做完了"，我会帮你生成接口规范文档。

---

> 等用户确认页面完成后，进入第二步。如果用户说"页面还没做"或"先帮我看看怎么做"，继续给出 impeccable 使用指导，不要进入第二步。

### 第二步：根据页面产出接口规范

用户确认页面完成后：

1. **读取前端代码**：找到页面组件文件（如 `src/pages/`、`src/components/`、`src/views/` 等），理解每个页面的：
   - 组件结构和数据流
   - 使用的字段名和数据类型
   - API 调用点（fetch/axios/api 调用）
   - 页面间的共享数据

2. **读取 PRD**：对照页面，确认是否覆盖了所有功能

3. **分析并标记问题**：在输出接口规范之前，先识别需要和用户讨论的问题，分两类：

   **不确定的问题**（需要用户确认）：
   - 页面上某个字段的具体含义或取值范围不明确
   - 某个交互的预期行为有多种理解
   - PRD 描述和页面实现有出入
   - 缺少必要的字段或接口

   **后端实现困难**（需要和用户讨论替代方案）：
   - 实时数据同步（WebSocket / 轮询的成本）
   - 复杂聚合查询（大量数据时的性能问题）
   - 第三方服务依赖（支付、短信、AI 推理等的延迟和成本）
   - 文件上传/处理（大文件、格式转换、存储成本）
   - 数据一致性（多表关联、并发操作）

4. **和用户讨论**：把发现的问题列出来，逐个和用户确认解决方案。格式：

   ```
   在分析页面和 PRD 后，发现以下需要确认的问题：

   **需要你确认的**：
   1. [问题描述] — 我的理解是 [方案 A]，你觉得对吗？
   2. [问题描述] — 有几种做法：[方案 A] vs [方案 B]，你倾向哪个？

   **后端实现上需要注意的**：
   3. [难点描述] — 这个实现成本较高，要不要 [简化方案]？
   4. [难点描述] — 建议 [替代方案]，你觉得可以吗？
   ```

   等用户逐一确认后，再输出接口规范。

5. **输出接口规范**到 `docs/plans/{prd名称}-{日期}-api-spec.md`

## 输出格式

文件命名：`docs/plans/{prd名称}-{日期}-api-spec.md`

```markdown
# [项目名] 接口规范

> 本文件基于已完成的页面设计生成，作为前后端开发的契约。

## 接口列表

### GET /api/todos
**用途**：获取 Todo 列表
**调用页面**：Todo 列表页

**响应**：
```json
{
  "todos": [
    {
      "id": 1,
      "title": "买菜",
      "completed": false,
      "createdAt": "2026-05-10T10:00:00Z"
    }
  ]
}
```

### POST /api/todos
**用途**：创建新 Todo
**调用页面**：Todo 列表页（添加表单）

**请求体**：
```json
{
  "title": "string (必填)"
}
```

**响应**：返回创建的 Todo 对象

---

## 数据依赖说明

| 页面 | 数据块 | 数据来源 | 后端成本 |
|------|--------|----------|----------|
| Todo 列表页 | todo 列表 | 数据库查询 | 低（简单 CRUD） |
| 统计页 | 完成率图表 | 数据库聚合 | 中（需要聚合查询） |

## 前端路由

| 路由 | 页面 | 说明 |
|------|------|------|
| / | Todo 列表页 | 核心页面 |
| /stats | 统计页 | 后续阶段 |

## 待确认事项

- [ ] [需要和用户确认的问题]
```

## 输出后

告诉用户：

> 接口规范已输出到 `docs/plans/xxx-api-spec.md`。
> 现在可以用 `/dev-phase-planner` 制定开发计划了——它会读取 PRD + 技术选型 + 架构 + 这份接口规范，直接拆分开发阶段。

## 边界情况

- **PRD 不够详细**：基于已有信息做合理推断，在接口规范中标注"待确认"
- **项目没有前端界面**（纯后端 API 服务）：不适用本 skill
- **用户说页面已经做完了但不是用 impeccable 做的**：直接读取已有页面代码，跳到第二步
- **页面太多**（超过 5 个）：先聚焦核心流程的 2-3 个页面的接口，其余标记为"后续迭代"
