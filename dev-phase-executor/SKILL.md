# 开发计划执行器

执行 dev-phase-planner 生成的阶段开发计划，严格遵循 TDD 流程和软件工程最佳实践。

## 核心原则

**先理解，再执行。** 在写任何代码之前，必须完整阅读计划文档、理解当前项目状态、确认前置依赖已满足。

**TDD 不是可选项。** 每个任务都必须经历 Red -> Green -> Refactor 循环。跳过测试直接写实现是不可接受的。

**小步提交，可回滚。** 每个任务完成后立即 commit，确保任何一步出问题都能精确回滚。

## 工作流程

### Phase 0: 准备工作

1. **定位计划文档**
   - 在 `docs/plans/` 目录下查找计划文件
   - 如果用户指定了具体的 phase，找到对应的 MD 文件
   - 如果用户说"执行计划"，从总览文档开始，确认当前应该执行哪个 phase

2. **阅读并理解计划**
   - 完整阅读目标 phase 的 MD 文档
   - 理解：里程碑目标、验收标准、前置依赖、任务拆解、涉及文件
   - 检查"当前进度"部分，找到第一个 `[ ]` 未完成的任务

3. **检查前置条件**
   - 验证"前置依赖"中列出的 phase 是否已完成（检查对应文档的 checkbox）
   - 检查所需的外部服务/配置是否就绪
   - 如果依赖未满足，**立即停止并告知用户**，不要猜测或跳过

4. **了解项目现状**
   - 浏览项目目录结构
   - 如果是已有项目，阅读关键文件了解现有代码风格和模式
   - 确认测试框架、构建工具等基础设施已就绪

### Phase 1: 逐任务执行（TDD 循环）

对每个未完成的任务，执行以下步骤：

#### 1.1 理解任务

- 阅读任务的**目标**、**测试用例**、**实现要点**和**验收**说明
- 查看**涉及文件**表格，了解需要创建或修改哪些文件
- 如果任务依赖之前的任务结果，确认前序任务已完成

#### 1.2 Red — 写测试

- 根据计划中的测试用例名称，编写测试代码
- 测试应该**先于实现存在**——运行时应该全部失败（Red 状态）
- 遵循项目现有的测试命名约定
- 测试应覆盖：
  - 正常路径（happy path）
  - 边界条件（空值、空列表、极大值等）
  - 错误场景（无效输入、权限不足等）
- 运行测试，确认全部失败，记录失败信息

#### 1.3 Green — 写实现

- 编写最少的代码让测试通过
- 不要过度设计——先让测试变绿，干净的代码留给 Refactor 阶段
- 遵循项目现有的代码风格和模式
- 如果实现要点中指定了文件路径和方法，按照计划执行
- 运行测试，确认全部通过（Green 状态）

#### 1.4 Refactor — 重构（按需）

- 如果代码存在重复、命名混乱或结构问题，在测试保护下优化
- 如果任务简单、代码已经清晰，跳过此步直接提交
- 重构后再次运行测试，确保仍然全部通过
- 如果重构引入了新问题，回到 Green 状态修复

#### 1.5 提交

- 将任务的 `[ ]` 更新为 `[x]`
- **Git commit**：根据任务性质选择合适的前缀
  - `feat(scope): 新功能描述` — 新增功能
  - `fix(scope): 修复描述` — 修复已有 bug
  - `test(scope): 测试描述` — 仅补充测试
  - `refactor(scope): 重构描述` — 仅重构，无功能变化
  - scope 对应功能模块名称，只提交当前任务涉及的文件

#### 1.6 继续下一个任务

- 回到 1.1，处理下一个 `[ ]` 任务
- 如果用户指定了只执行某个 task，完成该 task 后停止

### Phase 2: 阶段收尾

当一个 phase 的所有任务都标记为 `[x]` 后：

1. **运行全部测试**
   - 执行完整的测试套件，确保没有回归
   - 如果有集成测试或端到端测试，也一并执行

2. **请求用户验收**（仅当本 phase 有可手动验证的功能时）
   - 列出本 phase 的**验收标准**清单，请用户逐条操作验证
   - 根据用户反馈修改代码，通过的标记为 `[x]`
   - 如果本 phase 全部是纯后端/纯逻辑、无手动验证场景，跳过此步

3. **最终提交**
   - 如果收尾阶段有额外修改，做一个 `chore(scope): phase N 收尾` 的 commit

### Phase 3: 向用户报告

完成所有工作后，向用户报告：

```
## Phase N 执行完成

### 完成的任务
- [x] 任务 1: 简要描述
- [x] 任务 2: 简要描述
...

### 验收标准
- [x] 标准 1
- [x] 标准 2
...

### 测试结果
- 通过: X 个
- 失败: Y 个（如有，列出失败的测试）

### 已知限制
- 限制 1
- 限制 2
...

### 下一步
- 建议执行 Phase M（原因）
```

## 错误处理策略

### 测试失败

采用智能判断策略（计数粒度为单个 task）：

1. **分析失败原因**——阅读错误信息、堆栈跟踪、测试输出
2. **简单问题自动修复**（同一 task 内前 2 次失败）：
   - 拼写错误、导入路径问题、缺少依赖
   - 直接修复并重新运行测试
3. **复杂问题暂停求助**（同一 task 内第 3 次失败）：
   - 向用户报告：失败的测试、已尝试的修复、你的分析
   - 等待用户决定：继续重试、调整方案、或跳过该任务
4. **绝不修改测试来适应实现**——如果实现怎么都过不了测试，说明测试是对的，实现需要重新思考

### 计划与实际不符

执行过程中可能发现计划的实现要点有误（错误的文件路径、不存在的 API、过时的依赖）。此时：

- **暂停当前任务**，不要强行按错误的计划执行
- 向用户报告：计划写了什么、实际是什么、你建议怎么调整
- 等用户确认后再继续
- 如果调整影响后续任务，提醒用户可能需要更新计划文档

### 前置条件不满足

- 立即停止执行
- 明确告知用户缺少什么
- 等待用户解决后再继续

## 断点恢复

如果执行中断（用户切换任务、`/clear`、会话结束），恢复时：

1. 读取计划文档的"当前进度"，找到第一个 `[ ]` 任务
2. 检查该任务之前是否有未提交的代码变更（`git status`）
3. 如果有未提交变更，先判断是否与下一个任务相关——相关则继续，不相关则暂存并告知用户
4. 从断点处继续执行

## 执行纪律

- **不要扩大范围**——计划说什么就做什么，不要"顺手"重构不相关的代码
- **不要删除或修改已通过的测试**——除非计划明确要求
- **保持代码风格一致**——遵循项目现有的 linting 规则和格式化约定
