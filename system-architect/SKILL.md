# System Architect

根据 PRD 和技术选型文档，设计系统架构，输出可读性强的架构设计文档（Markdown + HTML），并与用户确认顶层架构共识。这份文档是后续脚手架搭建和开发计划的蓝图。

## 工作流程

### Phase 0: 读取上游文档

在项目根目录的 `docs/` 文件夹中查找上游文档：

1. **PRD 文档**：查找 `docs/*-PRD.md` 或 `docs/*PRD*.md` 或 `docs/prd.md`
2. **技术选型文档**：查找 `docs/*技术选型*.md` 或 `docs/*tech-stack*.md` 或 `docs/tech-stack.md`

如果找不到文档，询问用户提供文件路径或直接描述项目需求。

读取两份文档，提取关键信息：
- 从 PRD 提取：产品目标、核心功能、用户场景、非功能需求、约束
- 从技术选型提取：技术栈选择、已有架构草图、部署方案、成本约束

### Phase 1: 架构设计

按以下维度逐项与用户**深度讨论**。**每个维度开始前，先问用户是否需要设计这部分**。用户说"不需要"就跳过，进入下一个维度。

**讨论要求**：
- **你做架构，用户做评价** — 每个维度由你先给出完整方案和理由，用户只需要评价和补充场景/约束
- **不要让用户做设计决策** — 不问"你觉得用什么好"，而是说"我建议用 X，因为 Y"
- **用户输入的是场景和边界约束** — 追问时问"你的场景是什么"、"有什么限制条件"，而不是问"你想怎么做"
- **必须得出明确结论** — 用户确认或你根据反馈调整后，明确记录结论，再进入下一个维度

设计过程中遵循以下原则：
- 从高层到细节，先确定架构风格和组件边界，再深入每个模块
- 每个决策都要有理由 — 不是"用 X"而是"用 X 因为 Y"
- 复杂度要匹配项目规模 — 小项目不要过度设计
- 关键交互用 Mermaid 图表达，比纯文字更直观

#### 1.1 系统全景

确定架构风格（单体、模块化单体、微服务、Serverless、混合）并说明理由。绘制系统上下文图，展示系统与外部角色/系统的关系。

#### 1.2 模块划分

将系统拆分为核心模块/服务，明确每个模块的：
- 职责（做什么）
- 边界（不做什么）
- 依赖（依赖谁、被谁依赖）
- 技术栈（用什么实现）

#### 1.3 数据流设计

描述关键业务场景的数据流转路径，包括：
- 同步请求流（用户操作 → 前端 → API → 服务 → 数据库 → 响应）
- 异步处理流（事件/消息 → 队列 → 消费者 → 结果）
- 状态管理策略（前端状态、会话、缓存一致性）

#### 1.4 API 设计

设计核心 API 接口，包括：
- RESTful 路由规划或 GraphQL schema
- 核心端点列表（方法、路径、说明、关键参数）
- 认证方式（JWT、OAuth、API Key）
- 版本管理策略
- 错误响应规范

#### 1.5 数据模型

设计核心数据模型：
- ER 关系图（Mermaid erDiagram）
- 核心实体及其字段
- 索引策略
- 数据库选型的具体用法（哪种数据放哪种存储）

#### 1.6 部署架构

设计部署方案：
- 环境划分（dev / staging / prod）
- 容器化方案（Docker + 编排平台）
- 网络拓扑（负载均衡、服务发现、Ingress）
- CI/CD 流水线设计
- 基础设施成本估算

#### 1.7 安全设计

设计安全方案：
- 认证与鉴权（RBAC / ABAC）
- 数据传输加密（TLS）
- 数据存储加密（敏感字段）
- 输入验证与防注入
- API 限流与防滥用
- 合规要求（如适用）

#### 1.8 可观测性

设计监控方案：
- 日志规范（结构化日志、级别定义）
- 指标采集（业务指标、技术指标）
- 告警规则（什么情况触发、通知到哪）
- 链路追踪（分布式追踪方案）

#### 1.9 异常与容错

设计容错机制：
- 全局错误处理策略
- 服务降级方案
- 重试与幂等设计
- 熔断器配置
- 数据一致性保障（最终一致 vs 强一致）

#### 2.2 生成 HTML 文档

输出到 `docs/{产品名称}-架构设计.html`，要求：

- **自包含**：所有样式内联，不依赖外部 CSS 框架
- **Mermaid.js 渲染**：架构图、ER 图、流程图用 Mermaid 语法，通过 CDN 加载 mermaid.js 渲染
- **排版美观**：
  - 使用衬线/无衬线字体搭配，中文用系统字体
  - 合理的行高（1.6-1.8）、段间距、标题层级
  - 表格带斑马纹、hover 效果
  - 代码块带语法高亮背景
  - 决策卡片式布局（重要决策用卡片突出）
  - 响应式设计，移动端可读
- **目录导航**：顶部固定 TOC，点击跳转
- **配色方案**：参考 Anthropic 技术博客风格 — 暖白背景、深色文字、柔和的强调色

HTML 中的 Mermaid 图表示例格式：

```html
<pre class="mermaid">
graph TD
    A[用户] -->|HTTP| B[API Gateway]
    B --> C[认证服务]
    B --> D[业务服务]
    D --> E[数据库]
</pre>
<script src="https://cdn.jsdelivr.net/npm/mermaid/dist/mermaid.min.js"></script>
<script>mermaid.initialize({startOnLoad:true, theme:'neutral'});</script>
```

### Phase 3: 更新文档索引

在项目根目录的 `AGENTS.md` 的 `## 文档` 部分追加：

```markdown
- [系统架构设计](docs/{产品名称}-架构设计.md) — {产品名称}系统架构设计（含 [HTML 版](docs/{产品名称}-架构设计.html)）
```

如果 `AGENTS.md` 不存在，创建文件并包含 `## 文档` 部分。

### Phase 4: 确认与交接

向用户展示架构设计的关键决策摘要（不超过 10 条），询问：

1. 整体架构风格是否认可？
2. 模块划分是否合理？
3. 有没有需要调整的部分？

用户确认后，引导下一步：

> 系统架构设计已确认。下一步建议搭建项目脚手架——你可以说"帮我搭建项目脚手架"来启动 scaffold skill。
